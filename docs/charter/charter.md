## 2. Mission Statement / Project Goal

We aim to develop a responsive web application for the Computer Science department at UBC that streamlines the Teaching Assistant (TA) allocation and management process. The platform will support TA coordinators, instructors, and students in organizing TA requests, assignments, and historical data — without attempting to automate scheduling.

The application will allow:
- TA coordinators to monitor department-wide allocation needs and view historical assignment data  
- Instructors to easily submit TA requirements and track allocation for their courses  
- Students to create profiles, apply for courses, and track their TA assignments  

The goal is to make the system highly maintainable, user-friendly for first-time users, and optimized for desktop usage. By simplifying reporting, improving data entry efficiency, and supporting future use cases via flexible database design, the system will reduce administrative friction and improve transparency across all stakeholders.

## 5. Working Norms and Expectations

- **Communication Platform:** Discord  
- **Meetings:** Weekly virtual meetings (minimum 45 mins), additional ad hoc as needed  
- **Repo:** GitHub with a protected `main` branch. All work will be merged into a `development` branch first via pull requests before being promoted to `main`.  
- **Workflow:** Feature branches → PRs into `development` → Peer Review → Merge → Promote to `main` after testing  
- **Commit Messages:** Use conventional commits (`feat:`, `fix:`, `chore:`, etc.)  
- **Task Tracking:** GitHub Projects board with assignees, due dates, and labels
- **Testing Standards:** All new features should be tested before merging into `main`. Members are expected to write basic unit or integration tests where applicable and confirm functionality manually when automated testing isn't feasible.
### Documentation Standards

- Maintain a master `file-description.md` in the root directory that contains brief descriptions of all newly created or added files. Each entry should include:
  - File name
  - Purpose
  - Summary of functionality

- Provide inline documentation for:
  - Imported packages or libraries (especially non-standard ones)
  - New technologies or frameworks
  - Complicated logic, non-obvious syntax, or advanced language features

- Establish and maintain a consistent documentation tone that is **concise**.

- Goal: Make the codebase easier to understand, debug, and test across team members, especially when writing or reviewing tests.
